📝 Правила ведения блога проекта
Этот файл — руководство для тех, кто пишет посты в блоге проекта.
Здесь собраны правила тона, стиля и содержания.
Следование этим правилам делает блог живым, понятным и интересным для любого читателя.

1. Общие принципы
Блог ведётся ежедневно. Каждый день появляется минимум один пост.

Пост — это не технический отчёт. Это живой рассказ о том, что происходит в проекте.

Любой человек, даже далёкий от технологий, должен понять, о чём идёт речь.

Каждый пост готовится так, чтобы его можно было использовать в разных социальных сетях.

2. Тон и стиль
Пиши легко, по-человечески, иногда с иронией и самоиронией.

Избегай канцелярского, сухого, официального языка.

Можно быть честным: «это не получилось», «мы облажались», «неожиданно сработало».

Используй короткие предложения. Длинные — только когда без них не обойтись.

Можно задавать вопросы читателю, шутить, делиться эмоциями.

❌ Плохо:

«Был произведён рефакторинг модульной архитектуры с целью оптимизации производительности системы.»

✅ Хорошо:

«Переписали внутренности движка. Стало быстрее. Заняло два дня вместо запланированных четырёх часов — как обычно.»

3. Язык: никаких непонятных слов без объяснений
Запрещено использовать профессиональные термины и аббревиатуры без расшифровки.

Если термин необходим — сразу же объясни его простым языком в скобках или следующим предложением.

Пиши так, будто объясняешь другу, который умный, но не программист.

❌ Плохо:

«Настроили CI/CD пайплайн с деплоем через Docker в Kubernetes-кластер.»

✅ Хорошо:

«Настроили автоматическую сборку и выкладку (CI/CD — это когда код сам себя собирает и публикует без участия человека). Теперь новые версии появляются на сайте автоматически после каждого изменения в коде.»

4. Структура поста
Каждый пост должен иметь:

Заголовок — краткий, живой, отражающий суть дня.

Дата в формате ГГГГ-ММ-ДД.

Тело поста — что произошло, что сделали, что не получилось и почему.

Итог дня (необязательно, но желательно) — одна-две фразы: что важного из этого дня.

Варианты для соцсетей — адаптированные версии текста под разные платформы (см. раздел 5).

5. Посты и социальные сети
Каждый пост блога должен быть подготовлен в формате той площадки, где он будет публиковаться.

5.1. Обязательные площадки
Площадка	Стиль	Особенности
Сайт	Полный, развёрнутый	Основная версия поста, без ограничений по объёму
Telegram	Разговорный, живой	Допускаются эмодзи, призывы к обсуждению, форматирование жирным
MAX	Короткий, ёмкий	Максимальная концентрация смысла, можно разбить на несколько сообщений-нитью
VK	Близко к сайту	Чуть легче и визуальнее, допускаются эмодзи, абзацы, ссылки
5.2. Порядок работы
Сначала пишется основной текст — полная версия для сайта.

На его основе создаются версии для Telegram, MAX и VK.

Смысл во всех версиях совпадает — меняется только подача и объём.

Если текст для MAX приходится сильно сокращать — главное сохранить суть.

6. Формат записи в Blog.md
Каждый пост в файле Blog.md оформляется по следующей схеме:

text
# 2026-04-13 — Заголовок дня

[site]
Полная версия поста для сайта...

[telegram]
Версия поста для Telegram...

[max]
Короткая версия поста для MAX...

[vk]
Версия поста для VK...
Требования:

Все четыре блока ([site], [telegram], [max], [vk]) должны присутствовать у каждого поста.

Тон и смысл во всех версиях совпадают, но подача адаптирована под формат площадки.

7. Объяснение нового и сложного
Если в проекте появляется что-то новое — объясни это наглядно.

Используй аналогии из обычной жизни.

Сравнивай с чем-то понятным: «это как Google Maps, только для кода».

Если показываешь технический пример — обязательно поясни, что он делает.

8. Что писать
Темы для постов:

Что сделано сегодня (конкретно, без воды)

Какая проблема возникла и как её решили (или не решили)

Что изменилось в продукте с точки зрения пользователя

Интересная находка, инсайт, неожиданное открытие

Планы на следующий день

9. Чего избегать
Не пиши ради объёма — лучше короткий честный пост, чем длинная пустышка.

Не используй шаблонные фразы: «в рамках данного проекта», «с целью реализации», «на текущем этапе».

Не копируй содержимое технических задач или комментариев к коду без адаптации.

Не пиши в пассивном залоге там, где можно написать активно.

10. Оформление (Markdown)
Используй жирный текст для ключевых понятий при первом упоминании.

Используй код для имён файлов, команд, функций.

Используй списки, когда перечисляешь несколько пунктов.

Разделяй смысловые блоки пустой строкой.

11. Требования к интерфейсу программы
Чтобы работать с блогом было удобно, программа должна предоставлять:

Список постов — просмотр всех записей по дням.

Редактор постов — просмотр и редактирование каждого блока: [site], [telegram], [max], [vk].

Панель копирования — отдельные кнопки быстрого копирования для каждой площадки:

«Скопировать для Telegram»

«Скопировать для MAX»

«Скопировать для VK»

«Скопировать для сайта»

Автоматическая публикация — интеграция с API платформ для отправки поста напрямую из программы в одну или несколько соцсетей одновременно.

💡 Главный принцип: если ты прочитал пост и понял его — значит, он написан правильно.
Если пришлось перечитывать — переписывай.
